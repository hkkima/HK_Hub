## 1. 하드 리셋이 쉽게 만드는 것

기존 걱정(전 종목 슬리피지 없이 청산하며 AMM 총량보존 유지)이 사라진다. AMM을 어차피 새로 세우므로:

```
스냅샷 → 개인 정산 지급 → 주식 컬렉션 초기화 → 신규 4사 재상장(AMM 재시딩)
```

정교한 점진 마이그레이션 대신 **일괄 정산 + 초기화 + 재시딩**.

## 2. 리셋 대상 vs 보존 대상 (★가장 중요)

| 구분 | 대상 | 처리 |
|---|---|---|
| **초기화(wipe/reset)** | `stocks` · `holdings` · `trades` · `stockTraits` · `scheduledNews` · `meta/stockBoard`의 AMM 상태(시세·circulating 등) | 정산 후 삭제/리셋, 신규로 재시딩 |
| **보존(절대 유지)** | `users`(개인 balance — **정산 지급 받음**) · DP · 저지 · 보드 · 베팅 | 손대지 않음 |
| **★공유라서 주의** | **`housePool`(meta/stockBoard에 저장되지만 주식·DP·저지·봉사 4앱 공유)** | **주식 기여분만 정리, 나머지 앱 몫은 그대로** |

**핵심 함정: housePool은 공유 자원이다.** "주식장 리셋"이라고 housePool을 0으로 밀면 DP·저지·봉사 회계가 붕괴한다. → 리셋은 "공유 원장 안에서 주식 모듈만 청산·재시딩"이지 housePool 전체 wipe가 아니다.

## 3. 정산 정책 (평가가치 전액 보전)

사용자 확정: **보유주 × 현재가(mark value)를 전액 개인에게 환수, 매도 하락 미적용.**

회계적으로 mark value = **원가(cost basis) + 미실현손익(gains)** 로 분해된다:
- **원가 부분**: 매수 때 개인→housePool로 옮겨둔 몫 → housePool에서 개인으로 환원 (총량보존, 인플레 아님)
- **손익(gains) 부분**: mark value가 원가보다 큼(본딩커브 특성상 mark > 예치금). 이 초과분은 **신규 발행(mint)** 으로 지급 → 하드 리셋이 흡수하는 '리셋 보너스'

> 즉 "전액 보전"은 필연적으로 **gains만큼 순증(민팅)** 을 뜻한다. 리셋이니 수용 가능하지만, 인플레 규모를 사전에 계산·확인해야 한다(예산 편성 뷰·audit_house로).

**★선매수는 의도된 설계다(도덕적 해이 아님).** 평가가치 전액 보전 = "사도 손해 없다" 보장 → 배당 받으려 풀매수하는 것이 합리적. 명목 인플레는 **동기부여의 가격표**이며, 기업 레이어(금고 흡수)+R0 리베이스가 **현금 유출을 잡으므로 실질 인플레는 크지 않다**([CORP-POINTS.md](./CORP-POINTS.md) §8). 실측(선매수 상한): 명목 +60~171%, 현금 노출은 R0 리베이스+50% 흡수 시 현재 대비 ~1/6.

**housePool 처리**: 정산 후 housePool에서 '주식 기여분'을 뺀다(나머지=DP/저지/봉사 몫은 유지). 주식 기여분은 `audit_house.mjs`로 ledger type별 집계해 산출. 신규 시장은 별도 시드로 시작.

## 5. 신규 AMM 재초기화

- 신규 4사(리브랜딩 사명) 티커 신규 발급, 본딩커브 파라미터 초기값(base·slope·초기가·발행량) 재설정
- housePool 신규 시드: 새 시장이 필요로 하는 예치 시드만(나머지 앱 몫과 분리)
- **IPO 청약 자본금 분할**: 청약금 일부(예 20~30%)를 기업 금고로 → 개인 유동성 대량 흡수(경제모델 [CORP-POINTS.md](./CORP-POINTS.md) §8.1). reserve<mark 바이백 처리 결정 필요.
- **★DP 곡선 R0 리베이스 동시 수행**: `R0 10,000 → 25,000~30,000`(권장). 리셋 인플레로 DP가 상대적 헐값이 되는 것을 막는 현금 방파제. AMM 초기화 타이밍에 함께.
- **"새로운 컨셉" 여지**: 재상장 시 시장 모델 자체를 바꿀 수 있음. 미정 → §7 결정.

## 6. 안전장치 (되돌릴 수 없으므로)

- **백업**: 리셋 전 전체 Firestore export (단계 0).
- **드라이런**: 에뮬레이터/복제 프로젝트에서 정산·리셋 리허설 → 개인별 지급액·housePool 변화 검증.
- **정산 프리뷰**: 개인별 mark value 지급표를 운영자가 승인 후 실집행(자동 실행 금지).
- **close 윈도우 수행**: 금~일 거래 차단 중에만.
- **사후 재조정**: `audit_house.mjs`로 총량·housePool·앱별 몫 검증. 불일치 시 롤백(백업 복원).
